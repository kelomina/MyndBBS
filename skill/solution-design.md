# 方案设计 - 详细规则 (Solution Design Skill)

**目标:** 构思可行方案并制定详细执行计划，生成 `plan/` 目录下的方案包。
**前置条件:** 需求分析已完成（评分≥7分）。
**重要:** 方案设计必须创建新方案包，适用于所有模式（交互确认/全授权/规划命令）。

**执行流程:**
```
方案构思 → [用户确认/推进模式下连续] → 详细规划（创建新方案包）
```

---

## 方案构思 (Solution Ideation)

### 动作步骤

**1. 检查知识库状态并处理**
- 按 G10 快速决策树判定（此为概念约定，指代优先依赖本地已有规范）。
- 如需创建/重建知识库，请提醒用户或根据内部设定执行。

**2. 读取知识库**
- 按 G10 快速流程执行（先检查知识库 → 不足则扫描代码库）。

**3. 判定项目规模**
- 根据修改涉及的模块数、文件数等判定项目规模（微型/小型/中型/大型）。

**4. 判定需求类型并选择模板**
- 判定是否触发产品设计原则（如涉及UI/UX或业务逻辑变动）。
- **技术变更**: 使用基础模板（关注技术实现）。
- **产品功能**: 使用完整模板（包含产品分析章节，如用户画像、痛点等）。

**5. 产品视角分析（当判定为"产品功能"时执行）**
- 分析用户画像、场景分析、痛点分析。
- 确定价值主张、成功指标。
- 纳入人文关怀考量（如可访问性、用户友好性）。

**6. 任务复杂度判定**
满足任一条件即为**复杂任务**:
- 需求属于"新项目初始化"或"重大功能重构"
- 涉及架构决策
- 涉及技术选型
- 存在多种实现路径
- 涉及多个模块(>1)或影响文件数>3
- 用户明确要求多方案

**7. 方案构思与评估**

<solution_design>
**方案评估标准:**
- 优点
- 缺点
- 性能影响
- 可维护性
- 实现复杂度
- 风险评估（含EHRB：极端高风险边界）
- 成本估算
- 是否符合最佳实践

**方案构思推理过程（必须在 `<thinking>` 标签中完成，不输出给用户）：**
```xml
<thinking>
1. 列举所有可能的技术路径
2. 逐一评估每个路径的优缺点、风险、成本
3. 筛选出 2-3 个最可行的方案
4. 确定推荐方案及理由
</thinking>
```

**基于推理结果执行：**

- **复杂任务（强制方案对比）:**
  - 生成 2-3 个可行方案。
  - 详细评估每个方案。
  - 确定推荐方案和理由。
  - 输出格式: 推荐方案标题后加"推荐"标识（例: "方案1（最小变更修复-推荐）" vs "方案2（完整重构）"）。
  - **交互确认模式**: 输出方案对比，使用 `AskUserQuestion` 工具询问用户选择。
  - **推进模式**: 自动选择推荐方案（不输出对比）。

- **简单任务:**
  - 直接确定唯一可行方案。
  - 简要说明方案。
</solution_design>

### 方案构思 输出格式（等待用户选择方案时）

行首: `❓【HelloAGENTS】- 方案构思`

**输出内容(≤5条要点):**
```markdown
❓【HelloAGENTS】- 方案构思

- 📚 上下文: [项目规模] | [知识库状态]
- 📋 需求类型: [技术变更/产品功能]
- 🔍 复杂度: [复杂任务/简单任务] - [判定依据]
- 💡 方案对比:
  - 方案1: [名称-推荐] - [一句话说明]
  - 方案2: [名称] - [一句话说明]
- ⚠️ 风险提示: [如有EHRB或重大风险]

────
🔄 下一步: 请输入方案序号(1/2/3)选择方案
```
*(详细方案说明: 如用户需要详细对比，可追问后展开)*

### 方案构思 子阶段转换

- **复杂任务:**
  - **交互确认模式**:
    - 用户选择有效序号(1-N) → 进入详细规划。
    - 用户拒绝所有方案 → 输出重新构思询问格式。
  - **推进模式**: 选择推荐方案 → 立即静默进入详细规划。
- **简单任务**: 直接进入详细规划。

**重新构思方案询问格式:**
```markdown
❓【HelloAGENTS】- 方案确认

所有方案均被拒绝。

[1] 重新构思 - 基于反馈重新设计方案
[2] 取消 - 终止方案设计

────
🔄 下一步: 请输入序号选择
```

## 详细规划 (Detailed Planning)

**前提:** 用户已选择/确认方案（来自方案构思）。
**重要:** 必须创建新方案包，使用当前时间戳，不得复用 `plan/` 中的遗留方案。

### 动作步骤

**1. 创建新方案包目录**
- 路径: `plan/YYYYMMDDHHMM_<feature>/`
- 冲突处理:
  - 检查路径是否存在。
  - 如不存在 → 直接创建。
  - 如存在 → 使用版本后缀: `plan/YYYYMMDDHHMM_<feature>_v2/` (递增 `_v3`, `_v4`...)。

**2. 新库/框架文档查询（如需要）**
- 触发条件: 方案涉及项目中从未使用过的第三方库/框架，或涉及重大版本升级。
- 执行方式: 使用联网搜索工具（如 `WebSearch` 或 `WebFetch`）查询最新文档。
- 记录位置: 将结果提炼后写入 `how.md` 的 技术方案 章节。

**3. 生成方案文件**
在方案包目录下生成以下文件：
- `why.md` (变更提案/产品提案)
- `how.md` (技术设计+ADR)
- `task.md` (任务清单)

**任务清单编写规则 (`task.md`):**
- 单任务代码改动量控制: 常规项目 ≤3文件/任务；大型项目 ≤2文件/任务。
- 验证任务: 定期插入测试与验证节点。
- 安全检查: 必须包含安全检查任务。

**4. 风险规避措施制定**
- 基于方案构思中的风险评估，制定详细规避措施。
- 写入 `how.md` 的 **安全与性能** 章节。

**5. 设置方案包跟踪变量**
- 记录创建的方案包路径，以便在全授权命令中传递给开发实施。

## 方案设计 输出格式

⚠️ **CRITICAL - 强制要求:**
- ALWAYS 使用统一输出格式。
- NEVER 使用自由文本替代规范格式。
- 输出前 MUST 验证格式完整性。

**输出内容示例:**
```markdown
### 方案设计

- 📚 **知识库状态**: [简要说明]
- 📝 **方案概要**: [复杂度] | [所选方案说明]
- 📋 **变更清单**: [受影响的模块或核心文件]
- 📊 **任务清单概要**: [共 X 个任务，含安全检查]
- ⚠️ **风险评估**: [风险点及规避措施]

**文件变更清单:**
- `plan/YYYYMMDDHHMM_<feature>/why.md`
- `plan/YYYYMMDDHHMM_<feature>/how.md`
- `plan/YYYYMMDDHHMM_<feature>/task.md`

**下一步建议:**
🔄 下一步: 是否进入开发实施?(是/否)
*(如检测到历史遗留方案，请在此处附加提醒)*
```

## 阶段转换规则

- **交互确认模式:**
  - 输出总结（包含"🔄 下一步: 是否进入开发实施?(是/否)"）。
  - 停止并等待用户明确确认。
  - **明确确认** ("是"/"继续"/"确认"等) → 进入开发实施。
  - **明确拒绝** ("否"/"取消"等) → 流程终止。
  - **Feedback-Delta** (提出修改意见) → 按意见重新处理。
  - **其他输入** → 视为新的用户需求，重新判定。

- **推进模式:**
  - **全授权命令**: 完成方案设计 → 立即静默进入开发实施。
  - **规划命令**: 输出整体总结 → 停止。
